# Week 3 / Day 3 — Состояние задачи как конечный автомат (FSM)

Агент с формализованным состоянием задачи. Состояние описано как конечный
автомат и хранится в БД, поэтому работу можно поставить на паузу на любом
этапе и продолжить позже — без повторных объяснений.

---

## Конечный автомат

```
   ┌──────────┐   ┌───────────┐   ┌────────────┐   ┌──────┐
   │ planning │ → │ execution │ → │ validation │ → │ done │
   └──────────┘   └───────────┘   └────────────┘   └──────┘
                        ↑                │
                        └────────────────┘
                     (нашли проблему — назад)
```

| Этап | Что происходит |
|---|---|
| **planning** | формулируется план решения |
| **execution** | план выполняется |
| **validation** | результат проверяется |
| **done** | задача завершена (терминальное) |

Разрешённые переходы (`statemachine.py`):

```
planning   → execution
execution  → validation
validation → execution   (доработка)
validation → done
done       → ✗ (терминальное)
```

Любой другой переход **отклоняется** контроллером — даже если LLM его предложит.

### Состояние = тройка

- **stage** — текущий этап автомата
- **current_step** — что делается прямо сейчас
- **expected_action** — какое действие ожидается следующим

Плюс **status** (ортогонально этапу): `active` / `paused`.

---

## Как состояние влияет на агента

В `agent._make_system()` состояние кладётся **первым** в системный промпт:

```
## Состояние задачи
- Задача: Написать функцию валидации email
- Этап: Выполнение (execution)
- Текущий шаг: Предоставлен код и тесты, ожидание подтверждения
- Ожидаемое действие: Проверить код и перейти к validation или указать проблемы
- Маршрут: planning → execution → validation → done
```

После каждого хода контроллер (`_advance_state`) отдельным вызовом LLM решает,
как изменилось состояние, и применяет переход **только если он допустим**.

---

## Пауза и продолжение (проверено)

`python test_fsm.py` демонстрирует полный цикл:

1. старт → `planning`
2. утверждение плана → автомат переходит в `execution`
3. **⏸ пауза** на этапе execution — статус `paused`
4. во время паузы продвижение этапа **запрещено** (этап заморожен)
5. **симуляция перезапуска процесса** — создаётся новый экземпляр агента
   с пустым RAM; он восстанавливает этап / шаг / ожидаемое действие **из БД**
6. **▶ продолжение** — агент продолжает с текущего шага, **не переспрашивая**,
   что за задача (тест проверяет, что в ответе нет «какая задача?»)

Отдельно видно работу гарда: при возобновлении LLM попыталась прыгнуть сразу
в `done`, но контроллер разрешил только валидный переход `execution → validation`.

## API-маршруты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/task` | Состояние, история переходов, список этапов |
| POST | `/task/start` | Начать задачу `{task_name}` (этап planning) |
| POST | `/task/advance` | Ручной переход `{to_stage}` (с валидацией) |
| POST | `/task/pause` | Пауза (заморозить этап) |
| POST | `/task/resume` | Возобновить |
| POST | `/task/reset` | Сбросить состояние |
| POST | `/task/auto-advance` | Вкл/выкл авто-продвижение |
| POST | `/ask` | Запрос; ответ содержит `task_state` и `last_writes` (переходы) |
