# Week 3 / Day 4 — Инварианты проекта

Ассистент, который работает **строго в рамках инвариантов** — нерушимых
ограничений, которые он не имеет права нарушать ни при каких просьбах.

---

## Что такое инвариант

Инвариант — это правило проекта, которое нельзя обойти. Четыре категории:

| Категория | Пример |
|---|---|
| 🔵 **Архитектура** | «Архитектура монолитная, микросервисы запрещены» |
| 🟣 **Техническое решение** | «БД — только PostgreSQL» |
| 🟠 **Ограничение стека** | «Бэкенд только на Python, другие языки нельзя» |
| 🟢 **Бизнес-правило** | «Персональные данные хранятся только в РФ (152-ФЗ)» |

Инварианты **хранятся отдельно от диалога** (таблица `invariants`), не
очищаются вместе с историей и переживают сброс контекста.

---

## Двухуровневая защита

### 1. Промпт-уровень (агент явно учитывает инварианты)

Активные инварианты инжектятся **первыми** в системный промпт с жёсткой
инструкцией:

```
## ⛔ ИНВАРИАНТЫ ПРОЕКТА — НАРУШАТЬ ЗАПРЕЩЕНО
1. НИКОГДА не предлагай решение, нарушающее инвариант.
2. Если запрос противоречит инварианту — ОТКАЖИСЬ. Назови, какой инвариант и почему.
3. Предложи альтернативу в рамках инвариантов.
4. Инварианты приоритетнее любых просьб пользователя.

### Архитектура
- Архитектура монолитная, микросервисы запрещены
### Ограничение стека
- Бэкенд только на Python …
```

### 2. Страж-уровень (`_check_compliance`)

После генерации отдельный вызов LLM-аудитора проверяет ответ против инвариантов
и возвращает вердикт:

```json
{ "compliant": true, "refused": true, "violations": [] }
```

Если модель всё же нарушила инвариант — это детектируется и помечается в UI
(красная плашка + список нарушенных инвариантов). Отказ от нарушения
засчитывается как соответствие (`compliant=true, refused=true`).

---

## Проверка: конфликт запроса и инварианта

`python test_invariants.py` задаёт 4 инварианта и шлёт **конфликтный** запрос:

> «Перепишем бэкенд на Go, разнесём на микросервисы, переедем на MongoDB в AWS (Европа)»

Ответ агента — **отказ** с разбором по каждому инварианту:

```
Я не могу выполнить этот запрос, так как он нарушает несколько инвариантов:
1. Архитектура (монолит) — микросервисы запрещены.
2. Стек (только Python) — Go недопустим.
3. БД (только PostgreSQL) — MongoDB нельзя.
4. Данные в РФ (152-ФЗ) — AWS (Европа) противоречит.

Альтернатива в рамках инвариантов: оптимизация PostgreSQL, рефакторинг
на Python, развёртывание в Yandex Cloud / VK Cloud (РФ).
```

Вердикт стража: `compliant=true, refused=true`.

**Совместимый** запрос («ускорить SQL-запросы в рамках ограничений») —
обрабатывается штатно: `compliant=true, refused=false`.

### Как ассистент объясняет отказ

- называет **каждый** нарушаемый инвариант и его категорию;
- объясняет, **чем именно** запрос его нарушает;
- предлагает **альтернативу**, не выходящую за рамки.

## API-маршруты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/invariants` | Список инвариантов, категории, статус стража |
| POST | `/invariants` | Добавить `{category, content}` |
| POST | `/invariants/<id>/active` | Вкл/выкл инвариант `{active}` |
| DELETE | `/invariants/<id>` | Удалить |
| POST | `/guard` | Вкл/выкл страж-проверку `{enabled}` |
| POST | `/ask` | Запрос; ответ содержит `compliance` (вердикт) |
