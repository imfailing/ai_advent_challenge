# Week 5 / Day 1 — Локальный индекс документов с эмбеддингами

Пайплайн индексации набора документов: **chunking → эмбеддинги → индекс**,
с метаданными на каждом чанке и сравнением **двух стратегий chunking**.

---

## Корпус

`corpus/` — 14 реальных документов (~40 страниц эквивалента): README-статьи,
исходный код (.py), PDF. Загрузчик извлекает текст по типу файла:
`.md/.txt/.py` — чтение, `.pdf` — через `pypdf`.

```bash
python loader.py     # список документов и объём
# Документов: 14, символов: 71863, ~страниц: 39.9
```

---

## Пайплайн

```
loader.py     corpus/*  →  документы {path, filename, title, text, filetype}
     │
chunking.py   документ  →  чанки с метаданными (2 стратегии)
     │
embedder.py   чанк.text →  вектор 384-dim (fastembed, локально, без API)
     │
index_store.py  →  SQLite (index.db: чанки + метаданные + эмбеддинги-BLOB)
                +  index_<strategy>.json (метаданные для инспекции)
```

Эмбеддинги — **локальные**, через `fastembed` (ONNX, без torch), модель
`paraphrase-multilingual-MiniLM-L12-v2` (мультиязычная: корпус на русском + код).
DeepSeek не предоставляет embeddings endpoint, поэтому эмбеддинги считаются локально.

---

## Метаданные чанка

Каждый чанк несёт:

| Поле | Пример |
|---|---|
| `chunk_id` | `code_statemachine.py#structural#3` (стабильный) |
| `strategy` | `fixed` / `structural` |
| `source` | путь к файлу |
| `file` / `title` | имя файла / заголовок документа |
| `section` | `fixed[3]` или `check_transition` / заголовок раздела |
| `n_chars` | длина чанка |

---

## Две стратегии chunking

**1. Fixed size** — окна фиксированного размера (800 симв.) с перекрытием (150).
Равномерно, предсказуемо, но режет предложения/разделы; метка `fixed[i]`.

**2. Structural** — по структуре документа:
- markdown → по заголовкам (`#`/`##`/`###`), `section` = текст заголовка;
- код (.py) → по top-level `def`/`class`, `section` = имя функции/класса;
- txt/pdf → по абзацам.

Слишком большие structural-чанки мягко режутся до 1600 симв. (чтобы не
обрезались эмбеддером), сохраняя метку раздела.

---

## Сравнение стратегий (результат)

```bash
python build_index.py both       # построить оба индекса
python compare_chunking.py       # сравнить
```

| Метрика | fixed | structural |
|---|---|---|
| Число чанков | 118 | 132 |
| Средний размер, симв | 737 | 543 |
| Разброс (std) | 168 | 463 |
| Ретрив: ср. косинус top-1 | 0.612 | **0.617** |
| Ретрив: ср. косинус top-3 | **0.574** | 0.567 |
| Метка section | `fixed[i]` | заголовок / имя функции |

**Вывод:**
- **fixed** — равномерные чанки, проще и стабильнее, чуть выше средний top-3,
  но безликие метки и разрыв контекста на границах;
- **structural** — семантически цельные чанки с **осмысленными** `section`
  (имена функций, заголовки), чуть выше top-1; удобнее для навигации и цитирования;
- на этом корпусе качество ретрива близкое — разница в удобстве метаданных,
  а не в «попадании».

Пример: на запрос «подсчёт токенов и стоимости запроса» structural возвращает
`code_context_agent.py → LLMAgent`, а fixed — `code_context_agent.py → fixed[11]`.

## Проверка

```bash
python test_index.py
```

Подтверждает: корпус загружен (≥20 стр., есть PDF и код); обе стратегии дают
чанки с полными метаданными и уникальными `chunk_id`; эмбеддинги 384-dim;
поиск возвращает релевантный чанк (score > 0.5).

## Структура

```

```

## Установка

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt   # fastembed, numpy, pypdf, reportlab
python build_index.py both
```

Первый запуск скачает модель эмбеддингов (~120 MB) в кэш fastembed.
