# Week 5 / Day 2 — RAG-агент (с RAG / без RAG) + контрольные вопросы

Функция **вопрос → поиск релевантных чанков → объединение с вопросом →
запрос к LLM**, два режима ответа (с RAG / без RAG) и мини-набор из 10
контрольных вопросов для сравнения качества.

Строится на индексе из day 1 (`loader` / `chunking` / `embedder` /
`index_store` + `corpus/` скопированы сюда).

---

## RAG-функция (`rag.py`)

```
ask_with_rag(question):
    embed(question) → index_store.search(top-k)     # поиск чанков
    → собрать контекст с метками источников          # объединение
    → system «отвечай только по контексту, цитируй»
    → DeepSeek                                        # запрос к LLM
    → {answer, sources[]}

ask_plain(question):
    → DeepSeek без контекста                          # базовый режим
```

---

## Сравнение режимов (пример)

Вопрос: *«Что происходит при попытке перепрыгнуть этап задачи?»*

**Без RAG** (общий, неточный):
> Обычно это приводит к ошибке или пропуску логики… я не выдумываю.

**С RAG** (точный, со ссылками):
> Такой переход не существует в таблице переходов и отклоняется как
> «перепрыгивание этапа». Отклонённая попытка записывается (accepted=0)…
> **Источники:** code_statemachine.py, article_task_lifecycle.md

## Результат сравнения (`evaluate.py`)

```
Верных ответов (pass): без RAG 6/10, с RAG 9/10   (+3)
Source recall (нужный источник найден): 10/10
```

- Где модель **не знает** проектной специфики (save_to_file, категории
  инвариантов, Scheduler) — без RAG **0%**, с RAG **100%**.
- Где вопрос общий (транспорт stdio, стоимость по токенам) — оба режима верны.
- **Source recall 10/10** — нужный источник всегда попадал в найденные чанки.

**Честное ограничение (Q1):** «три стратегии» с RAG не сложилось — structural
chunking разнёс каждую стратегию в отдельную секцию-чанк, и запрос не поднял
все три в top-k (даже при k=6 нашлась только Sticky Facts). Вывод: качество RAG
зависит от гранулярности чанков и k — не «магия», а настраиваемый ретрив.
