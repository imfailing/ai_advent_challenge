# Week 5 / Day 3 — Улучшенный RAG: реранкинг + фильтр + query rewrite

Второй этап после поиска: **реранкинг cross-encoder'ом + фильтр по порогу
релевантности** и **query rewrite**. Сравнение качества без фильтра и с ним.

Строится на индексе day 1/2 (пайплайн + `corpus/` скопированы сюда).

---

## Конвейер

```
вопрос
  │  [query rewrite]  — LLM даёт 2–4 варианта запроса (recall↑)
  ▼
retrieve top-N=12 (bi-encoder, широко)      ← первый этап (поиск)
  │
  ▼
rerank (cross-encoder jina-v2-multilingual) ← ВТОРОЙ ЭТАП
  │  переоценка пар (запрос, чанк)
  ▼
фильтр по порогу (0.3) + min_keep=2 → top-K=4
  │
  ▼
контекст → LLM → ответ + источники
```

- **Первый этап** (bi-encoder, day 1/2): быстрый, но оценивает запрос и чанк
  по отдельности — топ по similarity может держать нерелевантные вводные секции.
- **Второй этап** (cross-encoder reranker): видит пару (запрос, чанк) целиком,
  точнее ранжирует; порог отсекает нерелевантное.
- **min_keep=2** — страховка: если порог отсёк всё, оставляем топ-2 лучших
  (не возвращаем пустой контекст).

---

## Настройки

| Параметр | Значение | Смысл |
|---|---|---|
| `top_n` | 12 | сколько достаём поиском **до** фильтрации |
| `top_k` | 4 | сколько оставляем **после** |
| `threshold` | 0.3 | порог отсечения (rerank-скор 0..1) |
| `min_keep` | 2 | гарантированный минимум чанков |

---

## Как второй этап меняет выдачу (пример)

Вопрос: *«Какие три стратегии управления контекстом реализованы?»*

**Без фильтра** (top-4 по similarity) — держит вводные секции, нужный чанк
со списком стратегий (`code_context_agent.py`, sim 0.476) **отсеян** →
ответ: *«конкретные стратегии не указаны»*. ✗

**С фильтром** (rewrite + rerank) — rewrite дал 4 варианта, найдено 17,
реранкер **поднял** нужный чанк (sim 0.509 → **rerank 0.702**) в топ →
ответ: *«Sliding Window, Sticky Facts, Branching»*. ✓

Именно cross-encoder переупорядочил кандидатов: bi-encoder ставил его 5-м,
reranker — 1-м.

---

## Сравнение конфигураций (`evaluate.py`, 10 вопросов)

| Конфигурация | pass | source recall | ср. до фильтра | ср. после |
|---|---|---|---|---|
| **A** plain (без фильтра, day 2) | 9/10 | 10/10 | 12.0 | 4.0 |
| **B** + rerank + порог | **10/10** | 10/10 | 12.0 | **3.0** |
| **C** + query rewrite | **10/10** | 10/10 | 20.6 | 3.0 |

**Вывод:**
- baseline проваливает «списковый» вопрос (три стратегии) — bi-encoder не
  поднимает нужный чанк в top-K;
- **реранкинг + порог** чинит это (10/10) и при этом **ужимает** контекст
  (4.0 → 3.0 чанка) — меньше шума, точнее ответ;
- **query rewrite** добавляет recall на перечислениях (мульти-запрос
  подтягивает разбросанные по секциям факты);
- `min_keep` не даёт порогу «убить» ответ на вопросах, где реранкер строг.

*(возможна небольшая вариативность прогонов из-за недетерминизма LLM в rewrite.)*

## Запуск

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
export DEEPSEEK_API_KEY="ваш_ключ"

python build_index.py structural   # индекс
python app.py "ваш вопрос"         # выдача без/с фильтром + отсеянные
python evaluate.py                 # сравнение A / B / C на 10 вопросах
```

Первый запуск скачает модель эмбеддингов и реранкер (cross-encoder) в кэш.
