# Week 5 / Day 5 — Мини-чат (Flask) с RAG + источниками + памятью задачи

Веб-чат на Flask, который хранит историю диалога, на каждый вопрос ищет
контекст в базе через RAG, отвечает с учётом найденного и **всегда выводит
источники**. Плюс **память задачи** (task state): цель, уточнения, ограничения,
термины — чтобы не терять цель в длинном диалоге.

Объединяет наработки: RAG-конвейер (week 5), память задачи (идея week 3),
Flask-чат (week 2). Порт **5006**.

---

## Как работает ход диалога

```
вопрос пользователя
  │  1. сохранить в историю (SQLite)
  ▼
  2. RAG-поиск: rewrite (с учётом истории) → retrieve top-N → rerank → фильтр
  │     если релевантность ниже порога → «не нашёл, уточните» (без выдумки)
  ▼
  3. ответ: система = роль + ПАМЯТЬ ЗАДАЧИ, затем история + контекст
  │     ответ всегда с источниками (файл → раздел)
  ▼
  4. обновить память задачи (цель/уточнения/ограничения/термины)
  ▼
  5. сохранить ответ ассистента
```

### Контекстуализация уточняющих вопросов

В чате вопросы часто ссылаются на предыдущий диалог: «а чем они отличаются?»,
«почему?», «расскажи про второй вариант». Сам по себе такой вопрос ничего не
находит (эмбеддинг бессмысленный, релевантность падает ниже порога → «не нашёл»).

Поэтому перед поиском вопрос **контекстуализируется по истории диалога**:
rewrite восстанавливает полный самостоятельный вопрос (разрешает «они» →
«три стратегии контекста», «второй вариант» → «Sticky Facts») и по нему идёт
поиск **и** реранкинг. Без этого приложение переставало находить контекст уже
на 2-м уточняющем вопросе.

## Память задачи (task state)

Хранится в SQLite (`task_memory`) и обновляется LLM после каждого хода:

| Поле | Что хранит |
|---|---|
| `goal` | цель диалога (устойчивая, из слов пользователя) |
| `clarifications` | что пользователь уже уточнил (накопительно) |
| `constraints` | зафиксированные ограничения |
| `terms` | зафиксированные термины/определения |

Показывается в правой панели чата. **Важно:** цель берётся из слов
пользователя, а не из примеров в документах — иначе пример «мобильное
приложение для доставки» из корпуса подменял бы реальную цель (эта ловушка
специально закрыта в промпте обновления памяти и в системном промпте).

## Проверка на 2 длинных сценариях (`test_scenarios.py`)

Два диалога по 10 сообщений:

**1. Выбор стратегии контекста** — цель «выбрать стратегию управления
контекстом для чат-бота», ограничения (бюджет токенов, длинные диалоги).
Итог: **10/10 ответов с источниками**, цель сохранена, ограничения накоплены.

**2. Построение MCP-интеграции** — цель «построить MCP-интеграцию», ограничение
(работа по расписанию 24/7). Итог: **8/8 внутрибазовых ответов с источниками**
(2 хода честно ушли в «не нашёл»), цель сохранена.

Проверяется, что к концу длинного диалога:
- ✅ цель **не потеряна** и не подменена примером из корпуса;
- ✅ ограничения зафиксированы;
- ✅ каждый содержательный ответ идёт **с источниками**.

Ключевой момент: на «напомни цель» ассистент отвечает из памяти задачи и
явно отличает пример из документов от реальной цели пользователя.

## Запуск

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
export DEEPSEEK_API_KEY="ваш_ключ"

python build_index.py structural   # построить индекс
python app.py                      # веб-чат на http://localhost:5006
python test_scenarios.py           # 2 длинных сценария
```

---

## Маршруты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/` | UI чата |
| POST | `/ask` | `{message}` → `{answer, sources, found, task_memory}` |
| GET | `/memory` | текущая память задачи |
| GET | `/history` | история диалога |
| POST | `/clear` | новый диалог (сброс истории и памяти) |
