# Week 3 / Day 1 — Модель памяти ассистента (memory layers)

Агент с **явной трёхслойной моделью памяти**. Информация разделена на три типа,
каждый хранится в отдельной таблице SQLite и инжектится в запрос по-разному.

---

## Три слоя памяти

```
┌─────────────────────────────────────────────────────────────┐
│  🔵 КРАТКОСРОЧНАЯ (short-term) — текущий диалог               │
│     таблица messages · окно последних 10 сообщений           │
│     уходит в запрос как сами reply-сообщения                 │
│     ⟲ очищается кнопкой «Очистить диалог»                     │
├─────────────────────────────────────────────────────────────┤
│  🟡 РАБОЧАЯ (working) — данные текущей задачи                  │
│     таблица working_memory · key-value, привязаны к задаче   │
│     инжектится в системный промпт                            │
│     ⟲ очищается при «Завершить задачу»                        │
├─────────────────────────────────────────────────────────────┤
│  🟢 ДОЛГОВРЕМЕННАЯ (long-term) — профиль / решения / знания    │
│     таблица long_term_memory · категории profile/decision/    │
│     knowledge · инжектится в системный промпт                │
│     ✓ переживает очистку диалога                              │
└─────────────────────────────────────────────────────────────┘
```

### Как слой попадает в запрос

```
system prompt:
  [базовая роль]
  ## Долговременная память
  ### Профиль пользователя
  - Имя: Сергей, роль: backend-разработчик, стек: Python
  ### Принятые решения
  - Выбрано потоковое чтение через генераторы…
  ## Рабочая память (текущая задача: парсер CSV)
  - target_size: 2GB
  - streaming_decision: confirmed

messages:
  [последние 10 сообщений диалога]   ← краткосрочная
```

---

## Что и куда сохраняется (маршрутизация)

Информация попадает в слои двумя путями:

**1. Автоматически.** После каждого хода отдельный вызов LLM-«маршрутизатора»
(`_route_memory`) анализирует обмен репликами и **явно** классифицирует новую
информацию по слоям, возвращая структурированный JSON:

```json
{
  "task": "парсер CSV",
  "working": {"target_size": "2GB", "streaming_decision": "confirmed"},
  "long_term": [
    {"category": "profile",  "content": "Имя: Сергей, стек: Python"},
    {"category": "decision", "content": "Потоковое чтение через генераторы"}
  ]
}
```

Каждая запись показывается в чате блоком «🧭 Маршрутизация памяти» —
видно, **что** именно и в **какой** слой ушло.

**2. Вручную.** В панели можно дописать/удалить запись в любом слое напрямую,
задать активную задачу, завершить задачу. Тумблер «Авто-маршрутизация»
отключает автоматический разбор.

---

## Влияние на ответы (проверено)

После полной очистки **краткосрочной** памяти (диалог = 0 сообщений) агент
по-прежнему корректно отвечает, опираясь на рабочую и долговременную память:

```
Вопрос: «Напомни, как меня зовут и на чём я пишу? Что мы решили по парсеру?»
Ответ:  «Вас зовут Сергей, вы backend-разработчик на Python.
         По парсеру решили: потоковое чтение через генераторы…»
```

Без слоёв памяти агент после очистки диалога не знал бы ответа — это и
демонстрирует разделение и влияние слоёв.

## API-маршруты

| Метод | Путь | Слой | Описание |
|---|---|---|---|
| POST | `/ask` | все | Запрос; в ответе `memory` со снимком всех слоёв и `last_writes` |
| POST | `/clear` | 🔵 | Очистить **только** диалог |
| GET | `/memory` | все | Снимок всех слоёв (для восстановления UI) |
| POST | `/memory/auto-route` | — | Вкл/выкл авто-маршрутизацию `{enabled}` |
| POST | `/memory/task` | 🟡 | Задать активную задачу `{task}` |
| POST | `/memory/working` | 🟡 | Добавить `{key, value}` |
| DELETE | `/memory/working/<key>` | 🟡 | Удалить ключ |
| DELETE | `/memory/working` | 🟡 | Завершить задачу (очистить рабочую) |
| POST | `/memory/long-term` | 🟢 | Добавить `{category, content}` |
| DELETE | `/memory/long-term/<id>` | 🟢 | Удалить запись |
