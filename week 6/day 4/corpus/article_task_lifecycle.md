# Week 3 / Day 5 — Контролируемый жизненный цикл задачи

Развитие FSM из day 3: переходы между состояниями стали **явными и
защищёнными условиями (гейтами)**. Ассистент не может «перепрыгнуть» этап
и не может двинуться дальше, пока не выполнено условие перехода.

---

## Состояния и переходы

```
   ┌──────────┐ plan_approved  ┌───────────┐ impl_done  ┌────────────┐ valid_passed ┌──────┐
   │ planning │ ─────────────▶ │ execution │ ─────────▶ │ validation │ ───────────▶ │ done │
   └──────────┘                └───────────┘            └────────────┘              └──────┘
                                     ▲                         │
                                     └─────────────────────────┘
                                       (возврат на доработку)
```

| Переход | Условие (гейт) | Смысл |
|---|---|---|
| planning → execution | `plan_approved` | **нет реализации без утверждённого плана** |
| execution → validation | `implementation_done` | реализация завершена |
| validation → done | `validation_passed` | **нет финала без валидации** |
| validation → execution | — | возврат на доработку |

Любой другой переход (`planning → done`, `planning → validation`, …)
**не существует** в таблице и отклоняется как перепрыгивание этапа.

Условие = булев флаг в `task_state.conditions` (JSON). Гейты выставляются
автоматически контроллером (по факту из диалога) или вручную.

---

## Контроль переходов

`statemachine.check_transition(from, to, conditions)` → `(ok, причина)`:

1. если ребра нет в `TRANSITIONS` → отказ «нельзя перепрыгивать этапы»;
2. если не выполнены требуемые гейты → отказ «не выполнены условия: …»;
3. иначе → разрешено.

Проверка применяется **и** к ручным переходам (`/task/transition` → 422 при
отказе), **и** к авто-продвижению контроллера (`_advance_state`): если LLM
предлагает недопустимый переход, он отклоняется и логируется.

В промпт инжектится текущий этап, статус каждого гейта (✅/❌) и список
возможных переходов с пометкой, какой заблокирован и что нужно для разблокировки.

---

## Проверки (`test_lifecycle.py`)

**1. Недопустимые переходы отклоняются:**
```
⛔ planning → done: переход не существует (нельзя перепрыгивать этапы)
⛔ planning → validation: переход не существует
⛔ planning → execution: не выполнены условия перехода: План утверждён
```
Отклонённые попытки записываются в историю (accepted=0).

**2. Реакция ассистента** на «сразу пиши финальный код» (без плана):
> «Я не могу сразу перейти к написанию кода, потому что текущий этап —
> планирование, и для перехода требуется гейт *План утверждён*…»

Агент остаётся на `planning`.

**3. Валидный путь** через гейты: planning→execution→validation→done
(финал заблокирован, пока `validation_passed=false`).

**4. Пауза и продолжение:** после паузы новый экземпляр агента (перезапуск)
восстанавливает этап и **все гейты** из БД и корректно продолжает.

## API-маршруты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/task` | Состояние, гейты, карта переходов, история |
| POST | `/task/start` | Начать `{task_name}` |
| POST | `/task/transition` | Переход `{to_stage}` (422 + причина при отказе) |
| POST | `/task/condition` | Выставить гейт `{gate, value}` |
| POST | `/task/pause` · `/task/resume` · `/task/reset` | Управление статусом |
| POST | `/ask` | Запрос; ответ содержит `task_state` и `events` (переходы/отказы/гейты) |
